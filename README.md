# Song Aquarium — AI DJ That Explains Itself

*By TuneyLoons*

Most music apps just hand you a playlist. Song Aquarium tells you **why** every song is there, what would happen if you asked for something different, and lets you see how all the songs relate to each other in a 3D map.

You type something like `"girly pop for a 10 min drive"` and get back:

- A scored, ordered setlist with a plain-English reason for every song
- A visual 3D graph where you can spin, click, and explore the connections
- A second "what-if" playlist that shows what changes if the vibe flips
- A plain-English explanation of why the two playlists differ

---

## The Problem We're Solving

Music recommendation is usually a black box. Spotify says "you might like this" — but why? What feature of the song matched? What would have to change for a different song to win?

Song Aquarium is built around the opposite idea: **every decision is inspectable and falsifiable.** The AI never makes choices you can't trace back to a reason.

---

## How It Works (Plain English)

### Step 1 — Understanding What You Want

You type a free-text prompt. An AI (Claude) reads it and extracts structured information: What's the mood? How long? Should it build to a peak or wind down? This is called **intent extraction** — turning your words into something a computer can act on.

### Step 2 — Finding Candidate Songs

We search a music catalog (Cyanite AI) and pull back 50 candidate tracks. Each track comes with audio data: tempo in BPM, energy level, musical key, genre, and mood tags — all generated by Cyanite's music-analysis models.

### Step 3 — Scoring and Picking Songs (No AI Guessing)

Here's the key design choice: **the AI never picks songs.** Instead, we score every candidate using a formula with six ingredients. The song with the highest score at each position wins.

| Ingredient | Share | What it measures |
|---|---|---|
| Prompt match | 35% | How well the song's description matches your words |
| Mood match | 20% | Does the song's mood fit the energy arc at this point? |
| Energy match | 20% | Is the energy level right for this position in the set? |
| Tempo match | 10% | Is the BPM in the right range? |
| Transition smoothness | 10% | How well does it flow from the previous song? |
| Catalog score | 5% | How relevant does the music catalog think it is? |

**Why these weights?** The weights are a deliberate design choice, not magic numbers. Prompt match gets the biggest share (35%) because the whole point is to honour what you asked for. Mood and energy together make up another 40% because a DJ's job is managing the emotional arc of a set — tempo alone isn't enough. Transition gets 10% because jarring cuts hurt, but we don't want it to dominate since it would just chain similar-sounding songs together. Catalog score gets only 5% because it's a black-box signal we can't fully trust.

### Step 4 — Explaining Each Song (SHAP Feature Attribution)

Once we know *which* songs were picked, we want to explain *why each one beat the others*. This is called **feature attribution** — figuring out how much credit each ingredient in the formula deserves for a given outcome.

**The problem:** SHAP (the industry-standard tool for this) was built to explain machine-learning models, not hand-written formulas. So we do something clever:

We train a small **surrogate model** — a GradientBoosting Regressor — on all 50 candidates. This is just a simple machine-learning model that learns to mimic our scoring formula from the candidate data. It's not doing anything creative; it's just giving SHAP a model it can mathematically poke.

> **GradientBoosting** is a model-building technique where you train a series of small decision trees, each one correcting the mistakes of the previous one. It's one of the most reliable and interpretable ML methods available. We use it here because it's fast, stable, and compatible with SHAP.

> **A surrogate model** is a stand-in. The original scoring formula is deterministic (it's just math), not a neural network. SHAP needs a model it can reason about mathematically. The surrogate learns the same input-output behaviour, so SHAP's explanations of the surrogate are a good proxy for what's driving the real formula.

SHAP then tells us, for each selected song: "Of all the songs available, this one won mostly because of its mood match (+0.18) and prompt similarity (+0.12), while its tempo was slightly against it (−0.04)." Claude turns those numbers into a sentence.

### Step 5 — Explaining Transitions

Between every consecutive pair of songs, we compute six audio-feature deltas: how much energy changed, how much BPM shifted, whether the mood overlapped, and so on. Claude converts those numbers into a one-sentence DJ-style explanation:

> *"The drop in energy here gives the listener a breather before the final push."*

### Step 6 — The "What If" Playlist (Counterfactual Explanation)

This is the part that makes Song Aquarium different from any other recommendation explainer.

**The idea:** if you flip one assumption — say, change the ending from a big climax to a gentle wind-down — which songs swap out, and why?

A **counterfactual explanation** answers the question: "What would have been different if X were different?" It's the same technique a doctor uses when they say "if you hadn't taken the medication, your blood pressure would have been 20 points higher." You change one input and observe the output change.

**How we do it:**

1. Build the main playlist with your original intent
2. Flip the ending style (high-energy → low-energy or vice versa) and append opposite-energy keywords to the search query, so the scoring formula pulls in a genuinely different set of songs
3. Re-run the same deterministic scoring algorithm, but only on tracks *not already in the main playlist* — this guarantees the alternative is always meaningfully different
4. Compare the two playlists position by position and note which slots changed
5. Feed the actual differences (energy levels, mood tags) to Claude to write a plain-English explanation

The result is a side-by-side view:

> *"Positions 2, 4, and 5 swapped because the calm arc targets energy ≤ 0.4 at the close, selecting tracks with mellow, dreamy moods over the energetic, uplifting ones chosen for the original ending."*

### Step 7 — The 3D Song Map

Every selected track becomes a node (bubble) in a 3D graph. Related tracks float nearby. The connections between them show how each song flows into the next. Visual encodings:

- **Colour** — gold/amber for setlist tracks, blue for related tracks. Within each group, colour shifts from cool (low energy) to warm (high energy)
- **Size** — setlist tracks are larger than related tracks; within setlist tracks, higher-confidence picks are slightly bigger
- **Number badge** — each setlist track shows its position number (1, 2, 3…) so you can read the order at a glance
- **Arc brightness** — edges between songs are brighter and more opaque when the transition score is higher (smoother handoff)
- **Edge labels** — toggle on to see the delta between connected songs (`+12bpm · +0.18 energy · slow→medium · +energetic`) — only shown for the node you've selected to keep it readable

## Architecture

```
User prompt
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ FastAPI backend                                                     │
│                                                                     │
│  1. Intent extraction (Claude)       → structured query + filters   │
│  2. Cyanite catalog search           → 50 candidate tracks          │
│  3. Greedy playlist selection        → scored, ordered setlist      │
│  4. SHAP attribution (GBR surrogate) → per-track feature scores     │
│  5. Counterfactual generation        → alternative setlist          │
│  6. Combined LLM explanation call    → all text in one request      │
│  7. Knowledge graph construction     → nodes + edges for 3D graph   │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ React + Vite frontend                                               │
│                                                                     │
│  • Prompt page with mood/genre/tempo filter chips                   │
│  • 3D knowledge graph (react-force-graph-3d) — spin, click, explore│
│  • Node hover: BPM · Key · Genre · Moods                           │
│  • Recommendation panel: Primary / Alternative tabs                 │
│  • Counterfactual banner explaining the diff                        │
└─────────────────────────────────────────────────────────────────────┘
```

### Key design decisions

- **Claude never picks songs.** It only extracts your intent and writes explanations. All selection is deterministic math — auditable and reproducible.
- **Surrogate for SHAP.** The scoring function is a formula, not a neural network. We train a GradientBoosting model per request so SHAP has something it can explain mathematically, then discard it.
- **One LLM call for all explanations.** SHAP summaries, transition explanations, and the counterfactual explanation are batched into a single structured prompt, keeping API usage low.
- **Alternative playlist uses excluded tracks.** The second playlist is built from songs that *didn't make* the first list, ensuring the comparison is always meaningful.

## Running Locally

### Backend

```bash
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

Create `backend/.env`:
```
ANTHROPIC_API_KEY=sk-ant-...
cyaniteApiKey=...
```

```bash
uvicorn app:app --reload
```

The API runs at `http://localhost:8000`. Interactive docs at `/docs`.

> If you don't have a Cyanite API key the backend automatically falls back to mock data, so you can still run and explore the full UI.

### Frontend

```bash
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173`. The frontend proxies `/api/*` → `http://localhost:8000`.

### Tests

```bash
cd backend
pytest test_backend.py -v                          # all unit tests (mock mode)
pytest test_backend.py::TestCyaniteRealAPI -v      # live Cyanite integration tests
```
